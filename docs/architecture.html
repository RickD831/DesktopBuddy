Desktop Buddy — how it's built

A 172×640 touchscreen on an ESP32-S3, fed by a Python companion on the PC. Two agents, one little screen. Claude gets the orange face, Codex CLI gets the blue one — whichever is idle steps aside so the busy one gets the full stage; if both are working at once, they share it. Here's every moving part.

The big picture

Data flows left to right: Anthropic's cloud and both agents' local files feed the companion, which condenses everything into one small JSON frame every 2 seconds and ships it to the display over whichever link is available. Codex needs no cloud API at all — its own session logs already carry OpenAI's rate-limit numbers.

Where each number comes from

Independent sources, each used for what it's uniquely good at. If one disappears, the others keep working.

One frame, every two seconds

Everything the buddy knows arrives as a single line of JSON — identical over USB and Bluetooth. The firmware never talks to the network; it just renders what it's told.

{ "t":"s",                          // status frame
  "claude":"working",               // → mood: focused eyes + typing dots
  "head":"Claude is working",       // headline text
  "msg":"Edit: main.cpp",           // scrolling detail (from hooks)
  "proj":"Desktop Buddy",           // active project, or "2 sessions: …"
  "ctx":32, "ctxt":"323k / 1.0M",  // context-window gauge
  "use": { "pct":56, "rst":"4h 19m",        // 5-hour block
           "w":38, "wm":61, "wrst":"Jul 12", // weekly bars
           "day":"41M tok · $12.40" },
  "cpu":31, "ram":62, "clk":"14:32", "date":"Wed Jul 09",

  "cdx": { "ctx":24, "ctxt":"63k / 258k",       // Codex context gauge
           "pct":3, "rst":"6d 20h",             // Codex rate-limit window
           "state":"working", "head":"Codex is working",
           "msg":"using wait", "proj":"my-app" },

  "med": { "st":"playing", "t":"Real Love Baby",       // now playing
           "a":"Father John Misty", "pos":143, "dur":221 } }

// album art rides separately: 60 paced chunks of base64 RGB565 pixels
{ "t":"art", "seq":0, "n":60, "d":"…" }

// buddy → PC:  {"t":"hello"} on boot · {"t":"pong"} · {"t":"pet"} when petted
//              {"t":"mc","cmd":"play|next|prev"} from the transport buttons
// 30 s of silence → companion reconnects · 30 s without frames → buddy sleeps

The six screens

Tap anywhere (outside the face and buttons) to cycle. Battery, charge state and the USB/Bluetooth link icon ride along in the corner of every screen.

Moods

The face is state, not decoration — each expression maps to a real signal. Long-press the face any time to pet the buddy (it says thanks). These six are Claude's; Codex gets a deliberately smaller set (below) since its session logs don't carry a "needs approval" signal the way Claude Code's hooks do.

idle
connected, nothing happening; blinks & looks around
working
hook fired; pupils focus, typing dots pulse
waiting
permission prompt is up — Claude needs you
happy
task finished (or you petted it)
error
something failed; eyebrows deploy
sleep
companion offline 30 s; backlight dims
groove
music on — the media-screen dancer sways, blinks & twirls; the main face bobs

Codex's moods — a deliberately smaller set

idle
connected, nothing happening
working
recent session-log activity (mtime-based, no hooks exist for Codex)
done
a task just finished — squinty smile

No waiting or error mood for Codex — its logs have no event type for "needs your approval", so guessing one would just be wrong more often than right. That's a scope cut, not an oversight.

Sharing one face slot

Claude and Codex never fight for space — update_dual_layout() in buddy_ui.cpp re-evaluates this on every status frame:

Bug that cost two rounds of hardware photos to find: apply_mood()/ apply_codex_mood() only ever reset eye/mouth size, never their alignment — position was assumed fixed-forever at construction. But the stacked view's scale_face_shapes() does move eyes/mouth inward as well as shrinking them. So going solo again must explicitly restore both size and position from that mood's canonical geometry — not just re-run the mood function and assume it's enough.

What each file does

PC side Python
  • buddy_companion.pyThe brain. ClaudeWatcher (transcripts → activity/context), CodexWatcher (tails ~/.codex/sessions — context/usage/activity, no API needed), UsageAPI (Claude's official limits + token refresh), UsageTracker (token math fallback), OtelReceiver (cost listener), MediaWatcher (now playing + album art + transport commands), and the two transports (USB serial, BLE) with heartbeat.
  • buddy_hook.pyFired by Claude Code on every tool use / prompt / notification / stop (registered in ~/.claude/settings.json). Writes one event line; never blocks Claude.
  • run_buddy.batManual launcher with a visible console.
  • start_buddy_hidden.vbsSilent launcher; a copy lives in shell:startup so it runs at login.
Firmware C++ / LVGL 9
  • main.cppSetup + loop: parses JSON frames from both links, reassembles album art, reads battery, answers pings, 30 s offline watchdog.
  • buddy_ui.cppAll six screens, both faces and their animations (blink, pupil wander, mood morphs, head-bob, the dancer), update_dual_layout() arbitrating the shared face slot, bars, toasts, pet + swipe gestures.
  • lvgl_port.cPanel + touch glue: QSPI at 40 MHz to the AXS15231B, chunked DMA flush, software rotation to landscape. Built on Waveshare's proven vendor driver (axs15231b/, touch/).
  • ble_link.cppBluetooth: Nordic-UART-style service advertising as ClaudeBuddy.
  • power_latch.cppBattery power control (diagram below).
  • lcd_bl_bsp/Backlight PWM — full when awake, dim in sleep.

The battery trick

The PWR button doesn't actually switch the battery — it only feeds power while held. An I/O-expander chip holds the real switch, and the firmware must grab it during boot. Miss that window (like our first firmware did) and the board dies the instant you let go.
